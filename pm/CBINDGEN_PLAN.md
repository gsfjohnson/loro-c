# Plan: Build Loro as a C/C++-consumable library (`loro-c`)

## Context

[Loro](https://github.com/loro-dev/loro) is a Rust CRDT library. There is **no official C or C++ binding** — the official [`loro-ffi`](https://github.com/loro-dev/loro-ffi) crate uses **UniFFI**, which targets Swift / Kotlin / Python / Ruby / React Native, and produces mangled scaffolding rather than a stable, idiomatic C public API.

To consume Loro from a C++ project, the cleanest path is a **purpose-built Rust→C ABI wrapper crate** that uses `cbindgen` to emit a real `loro.h`, paired with a hand-written C++ RAII header (`loro.hpp`) on top. The build is driven by **CMake + Corrosion**, so a single `cmake --build` builds the Rust staticlib and the C++ side together. The Loro Rust crate is pinned to a specific crates.io version (no submodule).

Per the user's choices, the v1 surface targets **full parity with `loro-ffi`** (LoroDoc + Text/Map/List/MovableList/Tree/Counter + events/subscriptions + Awareness + EphemeralStore + UndoManager + FractionalIndex + version vectors/frontiers + JSON Path). This is a sizeable surface (`loro-ffi` exposes ~23 interfaces); the work is staged into milestones M1–M5 below.

> **Update (post-M5):** M1–M5 landed a strong *subset*, not full parity — an audit against upstream [`loro.udl`](https://github.com/loro-dev/loro-ffi/blob/main/src/loro.udl) put real coverage at roughly **70%**. The "full parity" goal above is superseded by the staged remaining work in [GAPS_PLAN.md](GAPS_PLAN.md) (milestones G1–G6: rich text, cursors, JSON-update sync, diff/patch, structured values, and the doc-utility/attribution long tail). Rich text (G1) has since landed.

The end product: a CMake target `loro::loro` exposing `#include <loro/loro.hpp>`, statically linked, that any C++20 project can `target_link_libraries(... loro::loro)` against.

## Key design decisions

**Opaque handles.** Every Rust type is exposed as `struct LoroFoo;` (incomplete in C) plus `LoroFoo* loro_foo_new(...)` / `void loro_foo_free(LoroFoo*)`. Internally these are `Box<Foo>` raw pointers.

**Container handle lifetime — strong co-ownership (verified against `loro-internal` v1.13.1).** Each container handle (`LoroText*`, `LoroMap*`, …), including one extracted from a value via the `ValueOrContainer` split, is simply a `Box<loro::LoroText>` (etc.). No keep-alive wrapper or parent back-pointer is required, because loro's attached handlers already hold a **strong** reference all the way to the document state — there is no `Weak` in the chain:

```rust
pub struct LoroText { handler: TextHandler }              // public wrapper
enum MaybeDetached<T> { Detached(Arc<Mutex<…>>), Attached(BasicHandler) }
pub struct BasicHandler { id, container_idx, doc: LoroDoc } // holds the doc BY VALUE
struct LoroDoc   { inner: Arc<LoroDocInner> }              // Clone = shared strong Arc
struct LoroDocInner { state: Arc<LoroMutex<DocState>>, … } // strong, not Weak
```

Consequences for the C ABI:
- **Free in any order.** An attached container handle stays valid after the originating `LoroDoc*` is freed; `loro_text_free` etc. just drop the `Box`. There is **no use-after-free hazard from free ordering**.
- **Handles keep the whole doc alive.** Because every handle is a strong co-owner, the doc's `DocState` is released only when the `LoroDoc*` **and every** extracted container handle have been freed. Leaking a container handle leaks the entire document — this must be documented in `loro.h`/`loro.hpp`.
- **C++ ergonomics (optional).** The C++ RAII layer *may* store a `std::shared_ptr<Doc>` inside each child wrapper to make the parent relationship explicit, but this is now purely cosmetic — it is **not** needed for memory safety.

This matches how `loro-ffi` (UniFFI) wraps the same handlers, so the design is consistent with upstream.

**Error model.** Every fallible function returns a `LoroStatus` enum (`LORO_OK`, `LORO_ERR_INVALID_ARG`, `LORO_ERR_DECODE`, ...). Detail messages are stored in a thread-local `last_error` and read via `loro_last_error_message()`. Rust panics are caught with `std::panic::catch_unwind` at every FFI entry (helper macro `ffi_guard!`).

**Strings & bytes.** Inputs as `(const uint8_t*, size_t)` for binary, `(const char*, size_t)` for UTF-8 (no null-termination assumed). Outputs as `LoroBytes { uint8_t* data; size_t len; size_t cap; }` with `loro_bytes_free`. The C++ wrapper hides this behind `std::string` / `std::vector<uint8_t>`.

**`LoroValue` marshalling.** v1 round-trips primitives as **JSON bytes** (Loro already supports `to_json_value`). This keeps the FFI boundary tiny. Containers embedded in values are returned as opaque container handles via the existing `ValueOrContainer` split — i.e. `loro_value_get_container(LoroValue*) -> LoroContainer*`. The returned handle is an owned `Box<handler>` that the caller must `_free` (see *Container handle lifetime*). A structured tagged-union `LoroValue` can be added later if profiling shows JSON overhead matters.

**Callbacks.** All `loro-ffi` callback traits (`Subscriber`, `LocalUpdateCallback`, `OnPush`, `OnPop`, `LocalEphemeralListener`, `EphemeralSubscriber`, `JsonPathSubscriber`, `FirstCommitFromPeerCallback`, `PreCommitCallback`, `ChangeAncestorsTraveler`) are marshalled as a uniform triple:
```c
typedef struct {
    void (*invoke)(const LoroDiffEvent* event, void* user_data);
    void* user_data;
    void (*free_user_data)(void* user_data);   // called when subscription dropped
} LoroSubscriberCallback;
```
Generic `CCallback<Args, Ret>` Rust struct holds the triple, implements the corresponding loro trait, and runs `free_user_data` in `Drop`. `Send + Sync` is asserted by contract — the C++ wrapper docs warn the user.

**Subscriptions.** `loro_doc_subscribe(doc, container_id, cb) -> LoroSubscription*`. Dropping the subscription handle unsubscribes (mirrors loro-ffi's `Unsubscriber`).

**Threading.** Loro itself is `Send + Sync`. The C ABI exposes nothing thread-affine; the C++ wrapper marks types as such via comments — no extra mutex.

## Build wiring (CMake + Corrosion)

```cmake
cmake_minimum_required(VERSION 3.22)
project(loro-c LANGUAGES C CXX VERSION 0.1.0)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

include(FetchContent)
FetchContent_Declare(
    Corrosion
    GIT_REPOSITORY https://github.com/corrosion-rs/corrosion.git
    GIT_TAG v0.5.1
)
FetchContent_MakeAvailable(Corrosion)

corrosion_import_crate(
    MANIFEST_PATH ${CMAKE_CURRENT_SOURCE_DIR}/loro-c-api/Cargo.toml
    CRATE_TYPES staticlib
    PROFILE release
)

# The committed include/loro/loro.h is the source of truth. It is NOT regenerated
# during a normal build — Cargo build scripts must not mutate the source tree, and
# regenerate-and-commit-on-every-build dirties the working tree in CI. Regenerate
# explicitly (opt-in target) only when the Rust API changes:
add_custom_target(regenerate-header
    COMMAND cbindgen --config ${CMAKE_CURRENT_SOURCE_DIR}/loro-c-api/cbindgen.toml
                     --crate loro-c-api
                     --output ${CMAKE_CURRENT_SOURCE_DIR}/include/loro/loro.h
    WORKING_DIRECTORY ${CMAKE_CURRENT_SOURCE_DIR}/loro-c-api
    COMMENT "Regenerating include/loro/loro.h via cbindgen CLI"
    VERBATIM
)

add_library(loro INTERFACE)
add_library(loro::loro ALIAS loro)
target_include_directories(loro INTERFACE
    ${CMAKE_CURRENT_SOURCE_DIR}/include
)
target_link_libraries(loro INTERFACE loro_c_api)  # Corrosion-imported target

if(BUILD_TESTING)
    enable_testing()
    add_subdirectory(tests)
endif()
```

`loro-c-api/Cargo.toml`:
```toml
[package]
name = "loro-c-api"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["staticlib", "cdylib"]

[dependencies]
loro = { version = "=1.13.1", features = ["counter", "jsonpath"] }
serde_json = "1"

# No [build-dependencies]: cbindgen is invoked as a pinned CLI tool, not from build.rs.
# CI installs it with: cargo install --locked cbindgen@0.29.4
```

**Header generation (explicit, not in `build.rs`).** There is no `build.rs`. The C header is produced by the **cbindgen CLI**, pinned to a known version, and invoked only on demand via the CMake `regenerate-header` target (or directly: `cbindgen --config loro-c-api/cbindgen.toml --crate loro-c-api --output include/loro/loro.h`). The generated `include/loro/loro.h` is **committed** and is the source of truth for IDEs and for any CI job that lacks a Rust toolchain.

Rationale: Cargo build scripts are only supposed to write into `OUT_DIR`. Writing the committed header from `build.rs` on every build dirties the working tree, can trigger Cargo rebuild loops under Corrosion, and makes CI non-deterministic.

**CI guard against drift.** A dedicated CI step regenerates the header and fails if it differs from the committed copy:
```bash
cbindgen --config loro-c-api/cbindgen.toml --crate loro-c-api --output include/loro/loro.h
git diff --exit-code -- include/loro/loro.h   # non-zero → header is stale, commit it
```
This verifies the committed header matches the Rust source without ever committing from inside a build.

## Risks & open questions

- **`loro` 1.x stability.** Pinning `=1.13.1` is exact; any minor bump may break the wrapper. Acceptable — matches `loro-ffi`'s own pinning practice.
- **Callback thread-safety.** Loro can fire subscribers from any thread that holds the doc. The user-supplied C callback must be reentrant. Documented in `loro.hpp` comments.
- **`cdylib` symbol versioning on Linux.** Still pending after the static-install M5 pass (see *M5 status* above): Corrosion-only, Linux/macOS-only, requires a SONAME baked via a rustc link-arg + a hand-built symlink chain, CI-only verification. The static `find_package`/`pkg-config` install landed without it.
- **Memory ownership of `LoroValue`.** JSON-bytes design avoids ambiguity; revisit only if profiling shows JSON serialization is hot.
- **Container handle lifetime — RESOLVED.** Verified against `loro-internal` v1.13.1: attached handlers hold a strong `Arc` chain to `DocState` (no `Weak`), so handles are strong co-owners. Handles are plain `Box<handler>`, free in any order, no use-after-free from free ordering (see *Container handle lifetime* under Key design decisions). Residual note, not a risk: a leaked container handle keeps the whole doc alive, so the leak-checking tests in M1 must cover freeing the doc *before* its child handles. Re-verify if the `loro` pin is ever bumped past 1.13.1.
